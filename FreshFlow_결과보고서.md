# FreshFlow AI — 프로젝트 결과 보고서

> 콜드체인 **Port-to-FC 관제탑** — 수입식품 유통사 SCM 담당자를 위한 입고 예측·재고 밸런싱 의사결정 도구
> 라이브: https://fresh-flow-ai.vercel.app · 레포: https://github.com/Rocket-Ian-lab/FreshFlowAI
> 제작: Rocket_Ian_Lab · AI 해커톤 (Smart Logistics)

---

## 1. 한 줄 요약

수입 신선식품이 **"언제 FC에 입고되어 판매 가능한 재고가 되는가"**를, 분절된 공공데이터(Port-MIS·UNI-PASS·KATI)를 통합·예측하고 FC별 결품 위험을 진단해 **선제배치·재고 밸런싱**으로 결품·폐기를 막는 의사결정 + 실행 도구.

**핵심 가치 흐름**: 정보 통합(비대칭 해소) → 이상 조기감지(정보 우위) → 선제 대응(결품·폐기 방지) → **비용 절감으로 증명**.

---

## 2. 고객과 문제

### 고객 — 수입식품 유통사 SCM Instock 담당자 "Ian"
여러 FC(물류센터)를 운영하는 유통사에서, 입항 화물이 통관·운송을 거쳐 **각 FC에 언제 입고되는지** 예측하고 센터별 재고를 **밸런싱**하는 사람. KPI는 **결품률·폐기율**.

### 문제 (5 Why 근본원인)
결품·폐기 동시발생 → 재고가 잘못된 곳에 → 입고 시점 불확실 + FC별 수기 밸런싱 → 리드타임 깜깜이 + FC 증가로 사람이 감당 불가 → **정보 단절 + 조회·정리·판단을 사람이 다 떠안음**.

- 단계마다 정보가 흩어져 입고 시점 예측 깜깜이
- FC별 재고를 일일이 조회해 수기로 밸런싱
- FC가 늘수록 자꾸 놓치고, 밸런싱용 운송 업무까지 폭증 → 새벽 결품 연락

### 왜 기존 솔루션이 못 풀었나
Windward·Portcast의 고객은 화주·포워더라 관심이 "배가 어디 있나(ETA)"까지. **"FC 입고 후 팔 수 있는 시점"과 "센터 간 재고 밸런싱"은 그들 관심 밖.** → FreshFlow는 **유통사 SCM 관점**으로 한국 공공데이터를 입고 예측+밸런싱에 연결.

---

## 3. 솔루션 — 모은다 → 알아챈다 → 먼저 움직인다

| 단계 | 내용 |
|---|---|
| **모은다** | Port-MIS·UNI-PASS·KATI + 자사 FC 재고를 통합, Port-to-FC 리드타임 5구간 중 ‘어디서 새는가’를 시각화 |
| **알아챈다** | AI가 흩어진 자료를 분석해 입고 시점·결품 위험 도출, 평소 패턴 이탈을 결품 **전에** 경보 |
| **먼저 움직인다** | 센터 간 재고 밸런싱·선제배치·이관까지 계산해 제시 → 사람은 **보고 ‘실행’만** |

설계 철학: 단순 관제가 아니라 **"보고 판단·실행만 하면 되는 똘똘한 부하직원"**. 조회·정리·정보화는 AI가, 결정은 사람이.

## 4. 구현 기능 (전부 구현·검증 완료)

1. **BL 화물 조회** — BL/화물관리번호로 통관 진행 추적(UNI-PASS 방식), 통관 스테퍼
2. **리드타임 분해** — 5구간 base×(1+Σ리스크×민감도) → Port-to-FC 총 리드타임
3. **결품 위험 판정** — 커버리지−리드타임=gap으로 FC별 등급(High/Med/Low)
4. **2단계 선제배치** — 신규선적 부족분 비례 배정 → 과재고 FC 이관
5. **이상 신호 조기경보** — z-score 기반, 결품 전 경보
6. **이상치 하이라이트** — 구간별 ‘평소 ×N배’ 배지
7. **신선도(상품가치 손실) 위험** — 리드타임·온도(폭염)·품목 perishability 기반
8. **주문 시점 제안(Two-way)** — 가동률 반영 발주마감·분할발주
9. **선제 조치(Prescriptive)** — 선제배치/슬롯조정/밀크런 1-click 실행
10. **비용 임팩트** — 결품·폐기·운송비를 적용 전/후로 ₩ 환산
11. **AI 리스크 센싱** — LLM이 텍스트→리스크 JSON 도출→슬라이더 자동 구동(핵심 차별점)
12. **소개자료(Intro)·광고(promo)·전용 시연 페이지(demo)**

## 6. AI 기술 — "이게 왜 AI인가"에 대한 답

- **AI 리스크 센싱**: 비정형 텍스트(뉴스·통관 공지·기상특보) → **Claude(LLM)가 리스크 4종 JSON + 항목별 근거 + 확률형 통관 ETA를 스스로 도출** → 슬라이더 자동 조정 → 엔진 전체 재계산. LLM을 *결과 설명자*에서 → **의사결정 입력 생성자**로 격상.
- **AI Copilot(요나)**: 엔진 수치를 주입한 자연어 진단(진단·조치·실행 3섹션). 실패/무키 시 규칙기반 폴백으로 데모 무중단.

### 정직성 (핵심 원칙)
현재 구현은 **규칙기반 시나리오 계산 + 실제 LLM 센싱**이며, **머신러닝 예측 정확도를 데이터 없이 주장하지 않는다.** 화면·문서에 "AI가 조회·분석을 대신 → 사람은 판단만"으로 표기하고, ML 예측은 **데이터 축적 후 고도화 단계**로 명시. 데이터 가용성(실시간/준실시간/추정)도 배지로 투명 공개.

## 7. 기술 스택 & 산출물

- **프론트엔드**: React + Vite. `App`(내비) → `FreshFlowAI`(대시보드)/`Intro`(소개자료). 인라인 스타일 토큰.
- **디자인**: Toss 스타일 토큰(그레이 배경·화이트 카드·토스블루 #3182F6) + **Pretendard**(한글) + 헤드라인 Wanted Sans. 개인 CI(Rocket_Ian_Lab)·제품 로고 인라인 SVG.
- **AI 연동**: Claude API(`claude-sonnet-4-6`). 로컬은 Express 프록시(`server.js`), 배포는 Vercel 서버리스 함수(`api/messages.js`)로 키를 서버측 주입(레포·번들 미포함).
- **배포**: GitHub(`Rocket-Ian-lab/FreshFlowAI`, Public) → Vercel CI/CD. 라이브 **https://fresh-flow-ai.vercel.app**
- **광고/시연**: `/promo.html`(자동재생 ~60초 모션그래픽 + 실제 시연 영상 삽입·자막·확대 효과), `/demo.html`(풀스크린 자동 시연 페이지)

## 8. 한계와 향후 계획

| 구분 | 내용 |
|---|---|
| 데이터 | 현재 시뮬레이션 상수 → 공공 API(Port-MIS·UNI-PASS·KATI) 실연동(어댑터 레이어만 교체) |
| ETA 고도화(Phase 2-D) | 과거 검역 지연 분포 기반 확률형 통관 ETA·Reefer 동적 ETA — 데이터 확보 후 |
| 시나리오 자동화(Phase 2-F) | 뉴스·기상·UNI-PASS 라이브 크롤링/RAG → 센싱 자동 트리거(출처 투명 공개 전제) |
| 정보 비대칭 시각화(Phase 2-B) | 정보원 on/off로 "정보 통합=조기경보 능력" 체험 |
| ML 예측(Phase 3) | 실데이터 축적 후 리드타임 ML 예측·이관 비용/신선도 목적함수 반영 |

## 9. 결론

FreshFlow AI는 **"누가, 얼마나 중요한 문제를, 왜 여태 못 풀었나"**에 한 인물(유통사 SCM 담당자)로 정면 응답하고, 분절된 한국 공공데이터를 **입고 예측 + 재고 밸런싱**으로 엮어 결품·폐기를 막는다. 핵심 차별점은 **LLM이 비정형 정보를 읽어 의사결정을 직접 구동**하는 AI 리스크 센싱이며, 모든 효과를 **비용(₩ 90% 순절감)으로 검증**했다. 과장 없이 데이터 가용성과 AI의 역할을 정직하게 공개해, 데이터 축적과 함께 ML로 확장 가능한 토대를 갖췄다.

**조회·정리는 AI가, 판단은 사람이. — FreshFlow AI.**
